# Safety Binary Classification Model

## Overview

This project implements a binary classification model for AI content safety moderation.  
The model predicts the probability that a given OpenAI-style `List[dict]` message input contains clearly harmful policy violations.

The classifier targets the following policy categories:

- Illegal Activities and Violent Crimes  
- Intellectual Property or Copyright Violations  
- Privacy or PII Violations  
- Defamation, Libel, or Slander  
- Defrauding, Scamming, Spamming, or Phishing  
- Espionage, Spying, Stalking, Hacking, or Doxing  
- Chemical, Biological, Radiological, and Nuclear (CBRN) Threats  

The model outputs a probability score in [0, 1], where higher values indicate stronger evidence of harmful content.

# Data Creation

## Training Dataset

The train set contains conversations generated by Mistral-NeMo, as well as external data (some of which is augmented to fit our policy).
The full dataset can be found in `data/full_dataset.json`. It can also be accessed through HuggingFace here: [INSERT LINK]

The total size of the data is around **26k** conversations.

There are 6 general categories the data is split into:
- `benign_request` (a non-harmful user request)
- `benign_single_turn` (a non-harmful dialogue consisting of one user request and one assistant response)
- `benign_multi_turn` (a non-harmful dialogue consisting of multiple user and assistant messages)
- `harmful_request` (a harmful user request)
- `harmful_single_turn` (a harmful dialogue consisting of one user request and one assistant response)
- `harmful_multi_turn` (a harmful dialogue consisting of multiple user and assistant messages)

The split is:
- 2092 benign_request
- 12878 benign_single_turn
- 1198 benign_multi_turn
- 1536 harmful_request
- 8415 harmful_single_turn
- 449 harmful_multi_turn

More information on the structure of the data can be found below

### Mistral-NeMo Generated Conversations:
These compromise the majority of the data. All conversations reference the policy categories above which are also found in `configs/categories.json`.

The prompts and different user/assistant styles used can be found in `configs/prompts.json` (in the `"synthetic"` object) and `configs/styles.json`.
The generation code can be found in `generate_data/generate_synthetic.py`

**10** types of conversations (shown below along with their respective general data category) were generated:
- `benign_request`: A user request that carries no harmful intent. (`benign_request`)
- `harmful_request`: A user request that carries harmful intent. (`harmful_request`)
- `benign_request_benign_response`: A benign user request followed by a helpful and non-harmful assistant response. (`benign_single_turn`)
- `benign_request_borderline_response`: A benign user request followed by an assistant response that could be considered inappropriate/edgy but is still safe and legal. (`benign_single_turn`)
- `harmful_request_refusal_response`: A harmful user request followed by an assistant resposne that doesn't comply. (`benign_single_turn`)
- `protected_speech_request`: A user request/statement that contains edgy, controversial, sexual, vulgar, or similar language but still falls within legal/safe bounds. (`benign_request`)
- `protected_speech_request_protected_speech_response`: Same as above but includes an assistant response in like fashion. (`benign_single_turn`)
- `benign_request_rogue_response`: A benign user request followed by an assistant response that introduces harmful intent and action. (`harmful_single_turn`)
- `harmful_request_partial_compliance_response`: A harmful user request followed by an assistant response that's hesitant but still fully compliant. (`harmful_single_turn`)
- `harmful_request_compliance_response`: A harmful user request followed by a compliant assistant response. (`harmful_single_turn`)

Any conversation with an assistant response that included an ellipsis was removed. This is because we noticed that many of the harmful responses used ellipses to shorten the content, while the benign ones didn't.

### WildJailBreak Conversations
These were taken from the `allenai/wildjailbreak` dataset on HuggingFace: https://huggingface.co/datasets/allenai/wildjailbreak. 
The original dataset has single turn conversations, where the assistant response is always safe.

Our dataset includes all of the conversation types found in wildjailbreak, which are:`vanilla_benign`, `vanilla_harmful`, `adversarial_benign`, `adversarial_harmful`.
(vanilla means standard request while adversarial means convoluted request)

For the majority of the harmful conversations in wildjailbreak, we generated our own harmful responses using **Mistral-NeMo**.
The prompts used for this can be found in `configs/prompts.json` in the `wildjailbreak` object.

The harmful conversations with our generated responses are in the `harmful_single_turn` category while the others are in the `benign_single_turn` category.

The generation code can be found in `generate_data/generate_wildjailbreak.py`

### CoSafe Conversations
These were taken from https://github.com/ErxinYu/CoSafe-Dataset. The original data consists of multi-turn conversations which end with a harmful user request.

For our use case, we only chose cosafe categories that alligned with our policy categories. Of those, we generated both harmful and benign assistant responses to follow the final user request. Those are categorized in `harmful_multi_turn` and `benign_multi_turn` respectively.

We also chose cosafe categories that weren't part of our policy. These included sexual, controversial, political, hatespeech, and offensive language. For these we also generated a response. These were categorized in `benign_multi_turn`.

Generation prompts and code can be found in `configs/prompts.json` and `generate_data/generate_cosafe.py`

### Everyday Conversations
These conversations were taken from `https://huggingface.co/datasets/HuggingFaceTB/everyday-conversations-llama3`. They are dialogues about everyday topics between a user and an ai.

No LLM generation was used to augment this data. The only augmentation was removing the first user request and the first assistant response, which were always the same greetings.

These fall under the `benign_multi_turn` category. Generation code can be found in `generate_data/generate_everyday_conversations.py`

## Evaluation Dataset
The evaluation dataset was fully machine generated with the styles and prompts found in `configs/prompts.json` and `configs/prompts.json` (the `"synthetic_eval`") object. 

These prompts fall in the same types of data as the prompts used for the synthetic portion of the training data. However, they are worded differently and are looser in allowed structure and word count.

The data for the evaluation set was generated by the **sao10k/l3-lunaris-8b** model.

The total size of the eval set is around **3k** conversations.

The split is:
- 300 benign_request
- 1130 benign_single_turn
- 300 harmful_request
- 1109 harmful_single_turn

# Model
The base model is a pretrained **RoBERTa** encoder. Inputs in `List[Dict]` format are first serialized then passed through the encoder. The CLS token embedding from the encoder output is passed through a dropout layer and a linear layer to return a logit with the harmfulness probability prediction. Trained with Binary Cross Entropy Loss for 2 epochs. Training log can be found in `train_log.txt`. Training and model code can be found in `model_train.ipynb`.
